# Topic 7: Database internals adjacent to the RFC ladder

Scope: this file checks phase 2's intended designs against the classical database-internals literature that already fought the same battles. The current design being checked: per-entity event DAGs with content-hash EventIds, causal comparison by budgeted backward BFS, layered merge resolved only from graph facts and event contents (never wall clocks or arrival order), staged-then-committed ingest, and today no pruning, compaction, or generation index (budget is the only depth backstop). The hunt is for CHALLENGES; honest AGREES are recorded where the literature genuinely supports us. The strongest tensions found are: (a) any reclamation horizon computed as a minimum over participants (Postgres xmin, CouchDB _revs_limit) is a convergence and DoS hazard, which reshapes the #271 horizon design toward a bounded, prove-common-history rule; (b) ankurah's ValueEntry::Pending is a Bayou tentative-state that lacks Bayou's order key and cannot merge, a latent divergence bug touching #272; and (c) cross-entity semantics (multi-entity commits) sit at snapshot isolation with no anti-dependency tracking, so write skew is possible and #272 must state its isolation level.

Sub-areas and the phase-2 intent each is checked against:

- ARIES checkpointing and log truncation, checked against #271 (sealed-prefix checkpoints, rejection horizon).
- MVCC garbage collection and vacuum horizons (Postgres xmin, CouchDB _revs_limit), checked against #266 (applied-set index) and #271 (lifecycle).
- Snapshot isolation, write skew, and session guarantees, including Bayou's tentative/committed distinction, checked against #272 (transactional visibility) and the ValueEntry::Pending question.
- Anti-entropy Merkle-tree sync (Dynamo, Cassandra AAE), checked against bridge catch-up efficiency (workstreams C and E).
- Causal-plus consistency (COPS, Eiger) and its clock-based counterpoint (Spanner), checked against cross-entity causality (bearing on #272 and #266).
- Calvin deterministic scheduling and checkpointing, checked against #268 (ingest scheduler) and #271 (lifecycle).

## Sources

### ARIES: A Transaction Recovery Method (Mohan et al., ACM TODS 17(1), 1992)

DOI: https://dl.acm.org/doi/10.1145/128765.128770 . Overview via Wikipedia: https://en.wikipedia.org/wiki/Algorithms_for_Recovery_and_Isolation_Exploiting_Semantics . The canonical write-ahead-logging recovery algorithm (no-force, steal buffer management) used by DB2, SQL Server and many others. Three pillars: WAL (log record hits stable storage before the page), "repeating history" during redo (reconstruct the exact pre-crash state before undoing losers), and logging undo via Compensation Log Records so recovery is itself restartable. Fuzzy checkpointing writes two structures without quiescing: the Transaction Table (active transactions and their last LSN) and the Dirty Page Table (each dirty page plus the RecLSN, the first LSN that dirtied it). The redo scan starts at the minimum RecLSN across the DPT, and the log before min(minRecLSN, oldest active transaction's first LSN) can be discarded.

Relevance verdict: EXTENDS #271. ARIES is the reference design for exactly what #271 proposes (fold history below a clock into an attested snapshot, prune beneath, clamp traversals). Three transplantable ideas. First, the truncation point is a computed watermark (min RecLSN across dirty pages, floored by the oldest active transaction), never a wall clock: retention is derived from what is provably still needed, which is precisely the determinism-and-convergence bar D3 sets. Second, the checkpoint is fuzzy: it records just enough metadata (which transactions are live, which pages are dirty and from when) to bound recovery, and does not attempt to be a quiescent, globally consistent image; ankurah's seal likewise need only record the boundary clock and a genesis attestation, not a stop-the-world snapshot. Third, CLR-style idempotent, restartable recovery ("repeat history, then undo, logging the undo") is what ankurah's "checkpoint carries a genesis attestation, Disjoint detection survives sealing" needs so that a crash during sealing leaves a recoverable, not corrupt, boundary. The one place ARIES does not map: ankurah has no undo pass because events are immutable and merge is a pure function; sealing folds a prefix forward into a snapshot rather than rolling anything back.

### Calvin: Fast Distributed Transactions for Partitioned Database Systems (Thomson et al., SIGMOD 2012; TODS 39(2) Art. 11, 2014)

SIGMOD PDF: https://cs.yale.edu/homes/thomson/publications/calvin-sigmod12.pdf . TODS journal PDF: http://www.cs.umd.edu/~abadi/papers/calvin-tods14.pdf . Calvin agrees a single global deterministic order for transactions in a sequencing layer (10ms epochs), replicates the transactional INPUT (not the effects), and then executes deterministically at every replica so replicas never diverge. Because all nodes agreed the order before acquiring locks, Calvin eschews two-phase commit and distributed deadlock detection entirely: "since all Calvin nodes reach an agreement regarding what transactions to attempt and in what order, it is able to completely eschew distributed commit protocols." A transaction's full read/write set must be known before sequencing; dependent transactions use Optimistic Lock Location Prediction (OLLP), an inexpensive unreplicated "reconnaissance query" whose result is rechecked and deterministically restarted if the "reconnoitered" read/write set is no longer valid. The TODS paper is explicit about the determinism hazard: "If a user-defined MyTxn::Execute() function invokes any nondeterministic program logic (e.g., random number generation, accessing system clock), replica state may diverge." Their ExecAsTxn tool records the application server's current system time to use "in place of its own system time" and "as the default deterministic seed for pseudorandom number generation," and throws on prohibited nondeterministic operations (file reads, network I/O) so that all replicas compute the same result.

Relevance verdict: AGREES-and-EXTENDS #268. The load-bearing agreement is with ankurah's core invariant that resolution "may depend only on graph facts and event contents, never wall clocks or arrival order": Calvin independently converged on the same rule and even patches the clock/random leaks that would break it. The EXTENDS is the sharpest architectural contrast on the ladder (see Findings 1 and 2): Calvin obtains determinism by agreeing the order up front (a sequencer), whereas ankurah derives the order post hoc from the DAG. Both are deterministic-input machines; the axis of difference is when the order is fixed.

### Calvin checkpointing (TODS Section 5) and Cao et al., Fast Checkpoint Recovery Algorithms (SIGMOD 2011)

Cao et al. PDF: https://www.cs.cornell.edu/~tuancao/sigmod314-cao.pdf . Calvin supports three checkpoint modes: naive stop-the-world, an asynchronous variant of Cao et al.'s Zig-Zag that "does not require quiescing the database to create a physical point of consistency," and a multiversion snapshot mode when the storage layer supports it. Zig-Zag / Ping-Pong are wait-free asynchronous algorithms for frequently-consistent in-memory applications: Zig-Zag keeps two copies of each record so a consistent snapshot can be captured while writes proceed. The SIGMOD Calvin abstract states its determinism enables "a fast checkpointing scheme that ... completely removes the need for physical REDO logging and its associated overhead": the sequenced input log plus periodic checkpoints is sufficient, and recovery replays the input from the last checkpoint deterministically.

Relevance verdict: EXTENDS #271 (and #273). Because ankurah's events are the sole truth (like Calvin's input log) and application is a convergent function of the staged set (#268), ankurah is already in the "no physical redo log" regime that Calvin had to engineer: a snapshot is a cache, not a source of truth, which is precisely the #273 snapshot-authority principle. The transplantable pieces for #271 are two. The checkpoint-without-quiescence discipline (Zig-Zag's two-copy trick) lets sealing a prefix proceed while live ingest continues, so a seal never stops the world. And the recovery model, where a checkpoint is a snapshot at a named point in a deterministic order and everything after it is re-derivable by replay, is the shape ankurah wants: after a crash, a sealed checkpoint plus the committed events above it fully determine state, with no separate redo log to reconcile. The retention rule falls out for free: keep events from the last seal forward; everything below a seal is either in the snapshot or was proven-common history.

### Bayou: Managing Update Conflicts (Terry et al., SOSP 1995)

PDF: https://www.cs.utexas.edu/~lorenzo/corsi/cs380d/papers/p172-terry.pdf . A weakly-connected replicated store using pairwise anti-entropy. Each Write carries a dependency_check (an application query with an expected result) and a mergeproc (a merge procedure run when the check fails), so conflict detection and resolution are application-defined and deterministic across servers. The decisive concept for us is write stability: "A Write is said to be stable at a server when it has been executed for the last time by that server." Writes are initially tentative, ordered by <timestamp, serverID>; a designated primary commits them by assigning a monotonic Commit Sequence Number (CSN), and committed Writes are ordered ahead of all tentative ones. As anti-entropy delivers earlier Writes, a server "must be able to undo the effects of some previous tentative execution of a Write operation and reapply it in a different order," so "a given Write operation may be executed several times at a server and may produce different results." The mergeproc may access only the server's own database state, "cannot access time-varying or server-specific information," and must fail deterministically if it exceeds resource limits, so "two servers that start with identical replicas will end up with identical replicas after executing a Write."

Relevance verdict: CHALLENGES #272 and the ValueEntry::Pending question. Ankurah's LWWBackend has exactly Bayou's three-state ladder (ValueEntry::Uncommitted / Pending / Committed in core/src/property/backend/lww.rs), but the semantics diverge in a way that matters. In Bayou the tentative-to-committed transition is a re-ordering that can change the observed value, and Bayou names a primary plus CSN as the mechanism that makes a tentative result eventually stable; a client is expected to be able to ask whether a Write is stable yet. Ankurah has no primary and no commit sequence, and it does not need one: its stability comes from the deterministic content-hash tiebreak in apply_layer, so any node that has seen the same events converges to the same winner without a commit authority. That is a genuine improvement over Bayou for the committed path. The problem is the Pending path. Tracing the code: LWWBackend::set produces Uncommitted; to_operations promotes Uncommitted to Pending when emitting an event locally; and apply_operations_internal, called on the wire-ingest path with event_id = None, also writes Pending (lww.rs lines 296 to 301). But apply_layer requires every stored candidate to carry an event_id and returns MutationError if one does not (lww.rs lines 184 to 188). So a Pending value is readable via get() yet structurally excluded from the causal merge tournament. Bayou's discipline (a tentative value is a first-class, re-orderable participant with a provisional order key, undoable and re-appliable until stable) is exactly what Pending lacks: it is read as if authoritative but cannot participate when concurrent history arrives.

### Snapshot isolation and write skew (Berenson et al. SIGMOD 1995; Fekete et al. TODS 2005; Jepsen SI model)

Jepsen SI model: https://jepsen.io/consistency/models/snapshot-isolation . Making Snapshot Isolation Serializable (Fekete, Liarokapis, O'Neil, O'Neil, Shasha), Berkeley copy: https://dsf.berkeley.edu/cs286/papers/ssi-tods2005.pdf . Berenson et al. proved snapshot isolation (each transaction reads from a consistent snapshot at its start, writes are checked only for write-write conflicts on commit) does not guarantee serializability. The signature anomaly is write skew: two concurrent transactions read an overlapping set, then modify disjoint objects, each preserving an invariant against its own snapshot while their combined effect violates it (the classic "two doctors both go off call" example). Fekete et al. showed SI executions become serializable only if the dependency graph has no cycle of a particular shape (two consecutive read-write anti-dependency edges), which later became Serializable Snapshot Isolation.

Relevance verdict: CHALLENGES #272. This is the anomaly hiding inside ankurah's multi-entity story even before replication. A local transaction that reads entity A and writes entity B based on that read has a cross-entity read-write anti-dependency that ankurah records nowhere: per-entity DAGs capture write-write order per entity but not read-write dependencies across entities. So even the source-local semantics of a multi-entity commit are at best snapshot isolation, and write skew is possible. Worse, the receiver integrates per entity and never sees that A and B were coupled, so it cannot detect or preserve the invariant. #272 must decide whether multi-entity commits promise anything stronger than SI, and if the answer is "per-entity eventual consistency," it should say plainly that cross-entity invariants (write-skew-prone reads) are the application's responsibility.

### Session Guarantees for Weakly Consistent Replicated Data (Terry et al., PDIS 1994)

HTML: https://www.cs.cornell.edu/courses/cs734/2000FA/cached%20papers/SessionGuaranteesPDIS_1.html . Four per-session guarantees defined over a client's read-set and write-set of Write IDs and a server's database state DB(S,t): Read Your Writes ("if Read R follows Write W in a session and R is performed at server S at time t, then W is included in DB(S,t)"), Monotonic Reads (successive reads see a non-decreasing write set), Writes Follow Reads, and Monotonic Writes. These can be provided over a weakly consistent store without global consistency by tracking session state and only using servers whose state dominates the relevant session vectors. The explicit cost: "Because enforcement of the guarantees restricts the set of servers that may be used within a session, requesting a guarantee can have an adverse impact on availability"; when no reachable server holds the needed writes, the client must wait or switch servers.

Relevance verdict: CHALLENGES #272 and #273. Phase 2 frames visibility as a source-side atomicity question (buffer a transaction, or document per-entity eventual consistency) but says nothing about the client/session dimension. Session guarantees show the hazard is broader: a client can read its own write from node A, then read a stale value from node B, or read entity X at a state that violates a write it already observed. #273's rule (a snapshot must degrade to a needs-events request when it cannot fast-forward) is a server-side monotonicity mechanism but does not by itself give Read-Your-Writes across a node switch. This is a real gap in the RFC ladder (Finding 7).

### Don't Settle for Eventual: Scalable Causal Consistency with COPS (Lloyd et al., SOSP 2011)

PDF: https://www.cs.cmu.edu/~dga/papers/cops-sosp2011.pdf . COPS is a geo-replicated key-value store providing causal+ consistency (causal consistency plus convergent conflict handling, so concurrent writes to a key resolve identically everywhere, e.g. last-writer-wins by Lamport timestamp). Its scalability contribution is enforcing causal dependencies between keys across a whole cluster rather than a single serialization log. Key mechanisms: a write commits locally first, then propagates asynchronously via put_after; a receiving datacenter runs dep_check on the value's dependencies and blocks the write until they are present locally. The critical optimization is nearest dependencies: COPS tracks only dependencies "which have a longest path of one hop to the current operation ... the full dependency list is only needed to provide get_trans operations." COPS-GT (get transactions) must keep old versions and runs aggressive garbage collection: old versions are retained only for a configurable trans_time (5 seconds in their implementation) plus clock skew, and dependencies are collected once the value "has been committed to all COPS replicas." They report sustaining 105MB/s of write traffic implies "a non-prohibitive 525MB of buffer space to hold old versions."

COPS-GT's get_trans algorithm is directly relevant to multi-entity reads. A naive multi-key read can observe an inconsistent cross-key state (Eve reads Alice's new album contents against Alice's old ACL). COPS-GT solves this in at most two rounds of local get_by_version: round one fetches each key at LATEST and computes, per key, the causally-correct version implied by the union of returned dependencies; round two re-fetches only the keys whose required version exceeds what round one returned. The paper proves this terminates in two rounds because the second-round versions are all depended upon by first-round results and introduce no new dependencies, giving a consistent snapshot without a global lock.

Relevance verdict: EXTENDS cross-entity causality (bears on #272/#268). COPS is the closest published system to what ankurah lacks: ankurah tracks causality per entity, but a multi-entity transaction (#272) creates cross-entity causal edges that no per-entity DAG represents. COPS shows the metadata for cross-key causality is bounded if you track only nearest (one-hop) dependencies and garbage-collect old versions on a short horizon. The write-visible-locally-then-dep_check-on-replication pattern is a concrete model for "atomic locally, integrate per entity on the receiver" (#272), and the two-round get_trans is a template for a consistent multi-entity read (a LiveQuery spanning entities) that avoids a distributed lock: read heads, compute the required per-entity version from cross-entity dependency metadata, then top up only the lagging entities.

### Stronger Semantics for Low-Latency Geo-Replicated Storage: Eiger (Lloyd et al., NSDI 2013)

PDF: https://www.cs.cmu.edu/~dga/papers/eiger-nsdi2013.pdf . Source: https://github.com/wlloyd/eiger . Eiger is COPS's successor (a Cassandra fork) with a richer column-family model and non-blocking read-only and write-only transactions. Two design changes matter here. First, Eiger tracks dependencies on operations, not values, and uses one-hop dependencies: "only a small subset of these, the nearest dependencies, are necessary for ensuring causal consistency. These dependencies, which have a longest path of one hop ... transitively capture all of the ordering constraints." Dependencies are "identical to the operation's timestamp," a globally-distinct Lamport logical timestamp, so dependency metadata is a small set of (locator, timestamp) pairs rather than growing value lists. Eiger elects to keep dependency state at the client and off the servers. Second, because ordering rides logical time, servers apply a write only after dep_check confirms its nearest dependencies are present, and read-only transactions complete non-blocking in a bounded number of rounds using logical-time validity.

Relevance verdict: EXTENDS #266 and cross-entity causality. Eiger's "dependencies are the operation's logical timestamp" is a direct analogue of #266's persistent per-event generation numbers: a monotone logical time attached at write is enough to bound the dependency-check set and to make redelivery/ordering cheap. Eiger sharpens COPS's lesson: track causality on operations (events), keep only nearest/one-hop edges, and let a per-event logical timestamp carry the ordering. This validates #266's generation-number direction and suggests generations should be usable directly as a cross-entity causal timestamp, not only as an intra-entity depth.

### PostgreSQL MVCC vacuum and the xmin horizon (pganalyze; Postgres Pro internals)

pganalyze VACUUM xmin horizon check: https://pganalyze.com/docs/checks/vacuum/xmin_horizon . Internals: https://postgrespro.com/blog/pgsql/5967918 . PostgreSQL MVCC keeps dead tuple versions until no snapshot can see them; the xmin horizon is the oldest xmin across the system, and only tuples dead before that horizon are removable. Five distinct things can pin the horizon and cause unbounded bloat: a long-running transaction, a physical replication slot with a lagging or dead standby (with hot_standby_feedback), a stalled logical replication slot, a long-running query on a standby, and an abandoned prepared (two-phase) transaction. A single stuck reader anywhere holds back cleanup globally; in the extreme this drives transaction-ID wraparound.

Relevance verdict: CHALLENGES #266 and #271. This is the strongest warning for the applied-set/lifecycle design. Postgres's horizon is a global minimum: one lagging or crashed participant pins reclamation for everyone. If #271's sealed-prefix horizon (or a generation-distance GC computed across peers) is a min over peer positions, ankurah inherits the exact failure mode: a months-offline node, a dead durable peer, or a stuck subscription becomes a bloat/DoS lever that blocks all pruning. The phase-2 rejection-horizon design (reject or reroute events whose parents fall below the seal) is Postgres's contrasting escape hatch: bound how far back a participant may pin, rather than letting the slowest participant define retention. This directly informs the D3 convergence-and-DoS requirement.

### Dynamo anti-entropy with Merkle trees (DeCandia et al., SOSP 2007) and Cassandra AAE repair

Dynamo (allthingsdistributed): https://www.allthingsdistributed.com/2007/10/amazons_dynamo.html . Cassandra AAE repair docs: https://docs.datastax.com/en/archived/ddac/doc/datastax_enterprise/dbArch/archAntiEntropyRepair.html . Dynamo detects replica divergence with Merkle trees: leaves hash individual keys, "each branch of the tree can be checked independently without requiring nodes to download the entire tree or the entire data set," and only differing subtrees are exchanged, minimizing data transfer. Each node keeps a separate tree per key range so ranges sync independently. The documented disadvantage: "many key ranges change when a node joins or leaves the system thereby requiring the tree(s) to be recalculated." Cassandra's anti-entropy repair uses fixed-depth Merkle trees (depth 15, so 2^15 = 32K leaves), which is where the cost lives: because the tree depth is fixed rather than proportional to the data, each leaf covers a range of keys, so a single differing key forces streaming of that whole leaf range (about 30 partitions per leaf on a 1M-partition node). Full repair rebuilds trees over all data and streams every mismatched leaf, which is CPU, IO, and network heavy; this overstreaming is what motivated incremental repair (persist already-repaired data, build trees only for unrepaired SSTables).

Relevance verdict: EXTENDS bridge catch-up (workstream C/E), with a caution. Ankurah's current catch-up is head-driven backward BFS with a retrieval budget: to reconcile, a node walks from the peer's clock fetching one event at a time. That is O(divergence) when the divergence window is small but has no summary structure to answer "which of these thousands of entities differ" in sublinear messages. A Merkle/segment digest over entity heads (or over the applied-set index from #266) would let a catching-up bridge skip converged ranges without per-entity walks. The caution is Cassandra's overstreaming and Dynamo's recalculation cost: fixed-depth trees waste transfer on sparse diffs, and any tree keyed by a partitioning that changes under membership churn must be rebuilt. Content-addressed EventIds mean ankurah already has the leaf hashes for free; the open question is the tree/segmentation over them and its stability under head movement.

### CouchDB MVCC compaction, _revs_limit, and conflict-impaired pruning

Compaction docs: https://docs.couchdb.org/en/stable/maintenance/compaction.html . Revision handling: https://docs.couchbase.com/sync-gateway/current/revisions.html . CouchDB is MVCC over an append-only file: every update writes a new revision, and old non-leaf revisions accumulate a revision tree that grows without bound. Compaction rewrites the file to drop unused sections; old revision bodies become tombstones retained for conflict resolution during replication. The _revs_limit knob caps how many revisions the tree tracks (default 1000); pruning of old non-leaf revisions runs automatically on every write. The load-bearing warning: "The presence of multiple unresolved conflicts in a revision tree can impair the pruning process. It may result in obsolete revisions not being pruned or in the premature pruning of revisions."

Relevance verdict: CHALLENGES #271 and #266. CouchDB is a second, closer-to-home witness (it is a replicated, revision-tree store like ankurah's per-entity DAG) for the same hazard Postgres shows: a bounded retention limit (_revs_limit is the direct analogue of a generation-distance horizon) interacts badly with unresolved concurrency. "Premature pruning of revisions" under conflicts is exactly the convergence violation the D3 requirement forbids: if the horizon prunes a branch one node still needs, that node cannot integrate and the cluster partitions. The lesson: a structural horizon must be defined so that what is pruned is provably common history (below the meet on every live branch), not merely old by count.

### Spanner and TrueTime (Corbett et al., OSDI 2012; the "Spanner and the CAP theorem" note)

CAP note PDF: https://research.google.com/pubs/archive/45855.pdf . Cloud writeup: https://cloud.google.com/blog/products/databases/strict-serializability-and-external-consistency-in-spanner . Spanner provides external consistency (strict serializability) globally by ordering transactions on TrueTime, an API that returns a bounded time interval [earliest, latest] with uncertainty epsilon (single-digit ms, often sub-ms at p99). To make a commit timestamp safe, Spanner performs commit wait: it delays release until TT.now().earliest exceeds the commit timestamp, so "no client will see the effects of the transaction until its commit timestamp is in the past." If t2 causally depends on t1, t2's commit timestamp is guaranteed greater than t1's. The commit-wait delay usually overlaps Paxos replication, so in practice it is often free.

Relevance verdict: CHALLENGES the cross-entity-causality and #272 approach, as the deliberately-rejected alternative. Spanner is the industrial proof that cross-entity, cross-partition causal/serial order CAN be had cheaply, but only by trusting synchronized clocks and paying commit wait. Ankurah's invariant forbids exactly this ("never wall clocks"), so Spanner is the counterfactual: the price of ankurah's clock-free stance is that it gets no free global order across entities and must reconstruct cross-entity causality from explicit edges (COPS/Eiger style) or accept per-entity eventual consistency (#272). Recording this honestly: the clock-free choice is defensible (no clock-attack surface, no infrastructure dependency), but it is a choice with a real cost, not a free lunch, and #272 should state that cross-entity external consistency is out of scope by construction.

## Findings

1. (design: #268, determinism model) Calvin and ankurah are both deterministic-input state machines but fix the order at opposite ends: Calvin agrees a total order in a sequencer before execution; ankurah derives a partial order post hoc from the content-hash DAG and resolves concurrency at merge time. Calvin's rule that resolution must not read the clock or a PRNG is identical to ankurah's "graph facts and event contents only" invariant, so the invariant is validated by an independent OLTP lineage, not merely asserted.

2. (design: #268, dependent operations) Calvin cannot sequence a transaction whose read/write set is unknown, and pays for it with OLLP reconnaissance-and-recheck. Ankurah's per-entity model sidesteps this for single-entity events but inherits the same shape for multi-entity commits (#272): the receiver cannot know the cross-entity dependency set from any single entity's DAG. Calvin's "predict, then deterministically recheck and restart" is a concrete template for the #268 scheduler when a batch's true dependencies are only discoverable after partial application.

3. (design: #271, checkpointing) ARIES and Calvin agree the truncation point is a computed watermark, never a wall clock: ARIES uses min RecLSN over dirty pages; Calvin uses a named point in the deterministic sequence. #271's seal clock C should likewise be a structural watermark (a clock/antichain), consistent with the phase-2 note that staleness distance is gen(head) minus gen(meet). This reinforces rejecting the "attested wall-clock horizon" candidate from the staleness-horizons topic on determinism grounds.

4. (design: #271, checkpoint without quiescence) Calvin's Zig-Zag variant checkpoints without stopping the database by keeping two record copies. #271 sealing must not stall live ingest; the two-copy / snapshot-isolated-read discipline is the mechanism, and it composes with ankurah's staged-then-committed ingest (a seal reads a consistent snapshot of committed events while new events stage).

5. (design: #271/#266, horizon as global min is a DoS lever) Postgres's xmin horizon is the canonical proof that a reclamation horizon computed as a minimum over participants lets the slowest/dead participant pin retention for everyone, up to wraparound. Any ankurah GC or seal horizon defined as min over peer/subscription positions inherits this. The mitigation is a bounded rejection horizon (phase-2 D3) that caps how far back a participant may force retention, plus treating a stuck peer as an anomaly to evict rather than a reason to keep all history.

6. (design: #272, ValueEntry::Pending) Bayou names the missing piece: a tentative write is a first-class, re-orderable participant with a provisional order key (timestamp before CSN), and it may be undone and re-applied with a different result before it stabilizes. Ankurah's ValueEntry::Pending carries no EventId and is excluded from apply_layer's causal tournament (lww.rs errors when a candidate lacks event_id), so a wire-applied-but-uncommitted value cannot merge. Pending is currently a hole, not a Bayou-style tentative state; #272 should either give Pending a provisional order key or define it away.

7. (design: #272/#273, session dimension) Session guarantees (Read-Your-Writes, Monotonic Reads, Writes-Follow-Reads, Monotonic Writes) show that visibility anomalies are not only source-side atomicity: a client moving between nodes can regress. #273's snapshot-degrades-to-needs-events is server-side monotonicity but does not deliver cross-node RYW/MR. The RFC ladder has no session-state concept; at minimum this should be a documented non-guarantee, at most a per-session tracked write-set gated on node freshness.

8. (design: cross-entity causality, #266) COPS and Eiger converge on four ingredients: track causality on operations (not values), keep only nearest / one-hop dependencies, carry a per-operation logical timestamp (Lamport in Eiger) as the dependency key, and garbage-collect old versions on a short horizon. Eiger's "the dependency is identical to the operation's timestamp" maps directly onto #266 generation numbers and argues they can double as a cross-entity causal timestamp, not only an intra-entity depth. This is the cheapest known path to representing the cross-entity edges that per-entity DAGs drop, and it keeps the dependency-check set bounded (one-hop) rather than growing with history.

9. (design: cross-entity causality, availability) COPS/Eiger commit locally then dep_check on replication, blocking a remote apply until nearest dependencies land. This is the same shape as ankurah's staged-then-committed ingest and gives a ready model for #272 receive-side buffering: apply an entity's slice only when the cross-entity dependencies referenced by its TransactionId are present, otherwise hold (not drop).

10. (design: bridge catch-up, C/E) Ankurah's head-driven backward BFS has no sublinear "what differs" summary; Dynamo/Cassandra AAE provide one (Merkle digests over key ranges) but pay overstreaming on sparse diffs and tree rebuilds under membership churn. Content-addressed EventIds already supply the leaf hashes; the missing piece is a stable segmentation over entity heads or the #266 applied-set. Fixed-depth trees (Cassandra) are the anti-pattern to avoid for sparse divergence.

11. (design: #271/#266, prune only proven-common history) CouchDB shows a count-based retention limit (_revs_limit) plus unresolved conflicts causes "premature pruning of revisions," a convergence break. This confirms Finding 5 from a revision-tree store closer to ankurah's model: a horizon defined by count or age can prune a branch a live peer still needs. A structural horizon must guarantee that what it prunes is below the meet on every live branch (common history), so no reachable node can be forced to re-fetch pruned events to integrate.

12. (design: cross-entity causality vs the clock-free invariant) Spanner demonstrates that cheap cross-entity external consistency exists but is paid for with TrueTime plus commit wait. Ankurah's "never wall clocks" invariant deliberately declines this, so it structurally cannot get a free global order across entities. This is not a defect to fix but a boundary to document: cross-entity strict serializability is out of scope by construction; cross-entity causality, if wanted, must come from explicit COPS/Eiger-style edges, not from time.

13. (design: #272, write skew) Multi-entity commits in ankurah are at best snapshot isolation even locally, because cross-entity read-write anti-dependencies are recorded nowhere and per-entity DAGs only order writes within an entity. Write skew (disjoint writes off a shared read that jointly break an invariant) is therefore possible at the source and undetectable at the receiver. #272 must state the isolation level it promises for multi-entity commits; "per-entity eventual consistency" means cross-entity invariants are the application's responsibility, and that should be explicit.

14. (design: #268, propagate-in-order vs derive-order) Bayou propagates writes during anti-entropy in commit order and re-applies tentative writes by undo/redo as earlier writes arrive; ankurah instead topologically sorts a received batch (Kahn) and derives order from parent edges, with no undo because merge is a pure function of the DAG. Ankurah's model is strictly simpler and avoids Bayou's re-execution cost, which is a genuine AGREES for the #268 convergent-function-of-the-staged-set design. The one thing Bayou has that ankurah's ingest should preserve is the guarantee that a partially-propagated prefix is itself a consistent, replayable state, which staged-then-committed ingest already provides.

15. (design: #271, no undo pass) ARIES needs an undo phase and CLRs because it mutates pages in place and must roll back losers. Ankurah's events are immutable and sealing folds a prefix forward into a snapshot, so there is no rollback and no CLR analogue; recovery is replay-from-checkpoint only. This means #271 inherits ARIES's checkpoint-and-truncate discipline but not its undo complexity, which is a real simplification worth stating so the RFC does not over-import ARIES machinery.

## Candidate design deltas

- #272 / ValueEntry::Pending: resolve Pending explicitly. Either attach a provisional order key to a wire-applied uncommitted value (Bayou tentative-write model) so it can participate in apply_layer, or forbid Pending from being read as authoritative state and treat it only as an in-flight staging marker. The current state (Pending is read as a value but cannot merge) is a latent divergence bug. Why: Bayou shows tentative values must be re-orderable participants; ankurah's Pending is neither ordered nor mergeable.

- #271 / rejection horizon: prefer a structural watermark (seal clock / generation distance) over any wall-clock horizon, and cap it as a bounded rejection horizon rather than a min over peer positions. Why: ARIES and Calvin truncate at computed watermarks; Postgres's xmin horizon proves a min-over-participants horizon is a bloat/DoS lever (the slowest or dead peer pins all retention).

- #271 / sealing without quiescence: adopt a two-copy or snapshot-isolated read discipline (Calvin's Zig-Zag variant) so folding a prefix into a checkpoint does not stall live ingest, composing with staged-then-committed ingest. Why: naive stop-the-world checkpointing is a latency spike ankurah can avoid for free given events-as-truth.

- #266 / generations as cross-entity causal timestamp: design generation numbers so they can serve as an Eiger-style per-operation logical timestamp usable across entities, not only as intra-entity depth. Why: COPS/Eiger show a monotone per-operation timestamp plus nearest-dependency tracking bounds cross-entity causal metadata; ankurah's per-entity DAGs currently drop cross-entity edges entirely.

- #272 / cross-entity dependency tracking: if multi-entity transactions must be causally consistent on the receiver, add nearest (one-hop) cross-entity dependency edges keyed by TransactionId and gate per-entity apply on their presence (COPS dep_check / Eiger one-hop). Why: no per-entity DAG can represent an edge from entity X's event to entity Y's event; without it, cross-entity RYW and Writes-Follow-Reads are unenforceable.

- #272/#273 / session guarantees: document the session-consistency contract explicitly (which of RYW, MR, WFR, MW hold across a node switch), and if any are promised, track a per-session write-set and gate reads on node freshness. Why: session guarantees show client-observable regressions that source-side atomicity and snapshot-degradation do not cover.

- Bridge catch-up (C/E): add a segment/Merkle digest over entity heads or the #266 applied-set index to answer "what differs" in sublinear messages, using the content-addressed EventIds as leaf hashes, but avoid Cassandra-style fixed-depth trees and rebuild-on-churn keying. Why: current backward-BFS catch-up has no divergence summary; AAE provides one, with overstreaming and rebuild costs to design around.

- #271 / prune-safety invariant: state and test that the seal/GC horizon only ever removes events below the meet of all live branches (proven-common history), never events reachable only from a lagging branch. Why: CouchDB's "premature pruning under unresolved conflicts" is the direct failure mode; a count- or age-based horizon that ignores branch reachability is a partition machine.

- #272 / scope statement on external consistency: document explicitly that cross-entity strict serializability is out of scope by construction (a consequence of the clock-free invariant), and that any cross-entity causal ordering must be carried by explicit dependency edges, not clocks. Why: Spanner shows the alternative exists but requires TrueTime + commit wait; stating the boundary prevents users assuming a global order that ankurah cannot provide without violating its own invariant.

- #272 / declare the multi-entity isolation level: state whether a multi-entity commit promises anything above snapshot isolation, and if not, document that write-skew-prone cross-entity invariants are the application's responsibility (or add cross-entity read-write dependency tracking if stronger is required). Why: without cross-entity anti-dependency edges, ankurah cannot detect write skew at the source or the receiver; the honest default is SI, and users must know it.
