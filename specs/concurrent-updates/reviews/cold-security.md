## Summary

- **Critical: 1** (EventBridge validation bypass)
- **High: 4** (creation policy bypass #243, unbounded bridge traversal #247, no message size limits #246, BFS fetch bypass #244)
- **Medium: 7** (incorrect chains #245, RwLock poisoning, DAG walk cost, bincode deser limits, cyclic parents, TOCTOU DoS, staging cross-contamination, head growth)
- **Low: 8** (downcast panic, backend name injection, event ID cross-validation, budget rate limit, lock contention, Clock sorting, entity_id mismatch, budget escalation)

### Key Observations

1. **Validation gap pattern**: The codebase has two distinct entry paths for remote events: (a) subscription updates via `apply_update` which validates, and (b) fetch/delta responses via `apply_delta` which does NOT validate in the EventBridge arm. This is the most critical finding.

2. **Resource exhaustion is partially mitigated**: The comparison budget (4000 max) bounds DAG traversal, but `collect_event_bridge` and message-level processing have no bounds.

3. **RwLock usage is inconsistent**: retrieval.rs uses poison recovery (`unwrap_or_else`), entity.rs uses `unwrap()`/`expect()`. A single panic in any backend path cascades through the entity.

4. **Deserialization is unbounded**: bincode deserializes with no size limits. Combined with the double-deserialization in LWW (outer `LWWDiff` then inner `data`), this is a viable OOM attack vector via crafted operations.

5. **Five of the 20 findings are already tracked** in GitHub issues #242-#247. The most critical finding (#1, EventBridge validation bypass) is only partially covered by #244 (which describes the CachedEventGetter path, not the apply_delta/EventBridge path).
